# Changelog

## v2.1.2 (2026-07-07)

### Features
- **VM: `match`, `break`, `continue`** тАФ the default bytecode path now compiles and executes value matching and loop control flow end-to-end (type checker, evaluator, compiler, VM)
- **VM: `x in list`** тАФ membership tests compile to a new `IN_LIST` opcode on the default path (no `FUNNY_INTERPRET=1` required)
- **Stdlib unification** тАФ VM and interpreter share `internal/stdlib.Call`; all 33 builtins (regex, env, file, http, crypto, jwt, sql, тАж) are available under `FUNNY_INTERPRET=1` as well as on the default VM path
- **Type checking: top-level expression statements** тАФ bare calls like `println(...)` at file top level are now type-checked (undefined functions and argument type errors are caught at compile time)

## v2.1.0 (2026-07-07)

### Features
- **F-string interpolation**: full `f"...{expr:spec}..."` support (lexer/parser/type checker/evaluator/bytecode VM) with a Python/Rust-flavored format-spec mini-language (`internal/strfmt`)
- **Real formatter**: AST-based `funny fmt` / MCP `format` tool (previously a no-op), preserving comments
- **Map literals**: `{key: value, ...}` construction, previously impossible (the type existed only as a `map[K, V]` annotation with no way to build a value)
- **Bracket line-continuation**: any `(...)`, `[...]`, or `{...}` may now span multiple lines; a newline inside an open bracket is insignificant whitespace, enabling the conventional one-entry-per-line-with-trailing-comma style for map/list literals and call arguments
- **`m[key]` indexing**: map values can now be read and written with bracket indexing (`m["a"]`, `m["a"] = 1`), in addition to `.field` access; index assignment also now works for lists (`xs[0] = 1`), via a new `SET_INDEX` bytecode instruction
- **Real module loading**: `import "path.fn"` (previously a pure no-op) now actually reads, parses, and merges the target file's `pub fn`/`struct` declarations, relative to the importing file's directory (`internal/module`). Unaliased imports merge `pub` symbols under their bare name (`add(1, 2)`); `import "path.fn" as m` keeps the module's own names unchanged and instead teaches call sites that `m.add(...)` means "call `add` from that module" (├а la Python's `import x as y`). Private (non-`pub`) functions remain reachable from within their own module but are hygienically renamed and inaccessible from outside it. Diamond dependencies are merged once; circular imports and cross-file symbol collisions are compile errors (`E1101`-`E1105`).
- **Plan engine step semantics**: `guard`, `delay`, retry `backoff`, and step `timeout` were AST-only labels that all executed identically to `tool` (see the M3 plan's own "branch is just a block" note) тАФ `internal/agent.Engine` now gives them real behavior. `guard` treats its body's final bare-expression/`return <value>` as a pass/fail assertion (`err(...)`/falsy fails, `ok(...)` always passes); `delay` requires `with timeout="<duration>"` and sleeps that long before running its body; `with retry max=N backoff=constant|linear|exp` adds a real delay between attempts (omitting `backoff` keeps the old immediate-retry behavior); `with timeout="<duration>"` bounds a single attempt's wall-clock time (documented caveat: the evaluator isn't preemptible, so a genuinely stuck step's goroutine isn't killed, only abandoned). Any step whose body ends in a bare expression/`return <value>` now also publishes that value to plan scope as `__result` тАФ previously documented in a doc-comment but never implemented. `branch` intentionally stays an alias for `tool` (no dedicated case-list grammar exists yet); `retry.on` (retry only for specific error types) stays deferred since Funny's errors are untyped strings. Fixed a latent data race (`internal/evaluator.Scope` wasn't safe for the concurrent access `parallel` steps already did from multiple goroutines) while adding this.
- **LSP server** (`funny lsp`, `internal/lsp`): a from-scratch LSP 3.17 implementation (no third-party protocol dependency), replacing the v1 scaffolding that was dropped in the v2 migration. Diagnostics (parser/module/type-checker errors, precisely positioned, with error codes), hover (types, function signatures, struct layouts, builtin/keyword docs), type-aware completion (locals/functions/structs/builtins/keywords generally, struct-fields-only right after `<expr>.`), signature help, go-to-definition (including across `import`ed files), document symbols (`fn`/`struct`/`plan`+`step` outline), formatting (delegates to the `funny fmt` formatter), find-references and rename (`textDocument/references`, `textDocument/prepareRename`/`rename`, scoped to the requested document, with function-level scoping for locals), and a custom `funny/planGraph` request that renders a `plan` block as a node/edge graph mirroring the plan engine's actual sequential/parallel execution semantics. See "LSP Server" in `docs/language-manual.md` for the full capability list and known scoping trade-offs.
- **Unified CLI**: LSP and MCP servers are `funny lsp` and `funny mcp` subcommands of the main `funny` binary (`go install github.com/jiejie-dev/funny/cmd/funny@latest`)

### Fixes
- **Struct field access was always typed as a string by the compiler, and builtin calls always returned `any`, regardless of what they actually returned.** `compileField` unconditionally reported every `a.b` access as `valStr`, so a struct field of any type other than `str` used in a typed operator (`p.x * p.x`, `item.price + tax`, `entry.count > 0`) failed to compile with a confusing `unsupported op * for str`; a `list[T]` function parameter/return annotation wasn't parsed at all, so looping over it produced an untyped loop variable with the same problem; and `internal/types.builtinReturnType` reported every non-`Result` builtin as `Primitive("any")`, which has no special-cased compatibility with concrete types, so e.g. `fn f() -> float: return sqrt(x)` failed with a spurious "expected float, got any" mismatch. Added struct-field-schema tracking to the compiler (`Compiler.structFields`, populated in a two-pass prescan so mutually-referencing structs work), taught `annotationValueType` to parse `list[T]`, and gave `builtinReturnType` the same concrete-type inference `builtinValueType` already had on the compiler side. As a side effect of routing builtin-call type-checking through the same path as user functions, builtin arguments are now actually visited by `CheckExpr` тАФ e.g. `sqrt(undefined_var)` now correctly fails with `E2001` at compile time instead of a confusing `vm: unsupported op LOAD_GLOBAL` at runtime
- **`for` loops silently skipped their first element under the default VM path.** `compileFor` initialized the loop index with a hardcoded `Emit(bytecode.PUSH_INT, 0)` тАФ an `Arg` of `0`, which the VM reads as "push `Constants[0]`", not the literal integer `0`. Since compiling the loop's iterable (e.g. a list literal) registers its own constants first, `Constants[0]` ended up holding whatever value happened to land there тАФ typically the iterable's own first element тАФ so the loop's index silently started at `1` instead of `0` on almost any real iterable. `for i in [1, 2, 3]: sum = sum + i` produced `5` (`2+3`), not `6`. This affected every `for` loop compiled by the default VM since the feature was introduced; only Go-level tests survived because the only existing test asserted on which opcodes were emitted, never on actually running the bytecode. Now uses `c.mod.AddConstant(0)` like every other integer literal
- **Half the documented standard library was uncallable from actual `.fn` source.** `regex_match`, `regex_replace`, `env_get`, `file_read`, `file_exists`, `http_get`, `md5`, `sha256`, `b64_encode`, `b64_decode`, `jwt_encode`, `jwt_decode`, and `sql_open` were fully implemented in `internal/vm/builtins.go` and documented in the language manual's builtin table, but were never added to `internal/types.builtinTypeNames` (the type checker's builtin allowlist) or `internal/compiler.builtinNames` (the compiler's CALL_BUILTIN-vs-CALL routing table). Every call to one of them failed at type-check time with `E2002 undefined function`, before the compiler or VM ever ran тАФ only Go-level VM tests that hand-built `bytecode.Module`s (bypassing the type checker and compiler entirely) ever exercised the implementations. Registered all thirteen in both allowlists, with the four that consistently return a `Result` on success and failure (`file_read`, `http_get`, `b64_decode`, `jwt_decode`) additionally recognized by the `?` operator. (`jwt_encode`/`sql_open` return a plain value on success and a `Result` only on failure тАФ an existing inconsistency in their VM implementation тАФ so they're deliberately left typed `any` rather than claiming a `Result` shape their success path doesn't have.) The tree-walking evaluator's builtin set (`internal/evaluator/builtin.go`, used only under the opt-in `FUNNY_INTERPRET=1` fallback) still only implements the original 8 M1 builtins and was not extended in this pass
- Every builtin call's result was statically typed `valNil` ("unknown") by the bytecode compiler regardless of what it actually returns, so using one directly as an operand of a typed operator тАФ `len(x) > 0`, `sqrt(x) < 1.0`, `str_contains(a, b) and c` тАФ failed to compile with `compileBinary: type mismatch nil vs X`, even though the identical source ran fine under `FUNNY_INTERPRET=1` (the tree-walking evaluator has no static value-type tracking to trip over). `compileCall` now infers a concrete type (`int`/`float`/`str`/`bool`) for the builtins that consistently return one
- The bytecode compiler had no opcode at all for `!=` on any type, `and`/`or` on `bool`, or `==`/`<`/`>`/`<=`/`>=` on `float` тАФ only `int`/`str`/`bool`/`nil` equality and `int` ordering compiled; `1.0 < 2.0`, `1 != 2`, and `true and false` all failed with `pickBinaryOp: unsupported op ...` under the default VM path despite working under `FUNNY_INTERPRET=1`. Added `EQ_FLOAT`, `LT_FLOAT`, `GT_FLOAT`, `LTE_FLOAT`, `GTE_FLOAT`, `AND_BOOL`, `OR_BOOL` bytecode instructions; `!=` reuses whichever `EQ_*` opcode `==` would pick and negates it rather than getting its own opcode family. `x in list` remains compiler-unsupported (VM-only, via the tree-walking evaluator) тАФ its operand-type rule (element vs. container, not same-type-both-sides) doesn't fit the existing `leftOp == rightOp` shape and was left as a follow-up rather than bolted on here
- `internal/evaluator.Scope`'s variable map was not safe for concurrent access, even though a plan's `parallel` step (`internal/agent.Engine.execParallel`) has always run its body's statements on separate goroutines against the same Scope тАФ a latent data race, caught while adding step `timeout` (which introduces a second source of concurrent Scope access: a timed-out step's abandoned goroutine can still be running). `Scope` is now internally synchronized
- `funny describe` was documented (README/language-manual/CLI help) and its underlying implementation (`cli.Describe`) existed and was unit-tested, but no `cobra.Command` in `cmd/funny/main.go` ever called it тАФ the subcommand didn't exist on the CLI at all. Wired it up alongside `run`/`ast`/`fmt`
- Same gap as `funny describe`, for `funny disasm`: `cli.Disasm` existed and was unit-tested, but had no CLI entry point
- Parser crash on standalone `#` comments (introduces `ast.CommentStmt`)
- Lexer bug where dedenting across multiple nesting levels to a non-zero column only emitted one DEDENT instead of all required levels
- Bytecode compiler crash (`unsupported statement type`) on any script containing a `meta:` or `plan "...":` block when run via the default VM path (`funny run` without `FUNNY_INTERPRET=1`) тАФ these are now no-ops in the compiler, matching the tree-walking evaluator
- Type checker rejected every struct-typed annotation (`let p: Point = ...`, `fn f(p: Point)`, `fn f() -> Point`, struct fields, `list[Point]`, ...) with a spurious type mismatch, because `ParseType` has no environment access and could only produce an opaque `Primitive("Point")` for a bare struct name instead of the real `Struct` type; type annotations are now resolved against the environment so struct names compare correctly everywhere
- Bytecode compiler crash (`vm: unsupported op LOAD_GLOBAL`) for any script that declares a top-level `fn` in between declaring and later referencing a top-level variable (e.g. `let p = ...` / `fn foo(): ...` / `println(p)`). `compileFnDecl` was resetting the compiler's local-scope table to a brand-new empty map after compiling each function body instead of saving/restoring the enclosing scope, permanently losing track of every local declared before that point; it also never isolated `varTypes` (local slot тЖТ value type) per function, so a variable and an unrelated function parameter sharing the same slot number could silently corrupt each other's recorded type and mis-codegen type-sensitive operators like `+`
- Lexer reported the wrong column (and byte offset) for every token except the first on a given line: `Position` was only captured once per line, during indentation handling, and silently reused for every later token on that line instead of being refreshed per-token. Every consumer that only ever looked at line numbers (error messages, `ast` output) was unaffected, but this made column-accurate tooling like the new LSP server impossible until fixed; caught while implementing hover/completion/go-to-definition, which need real per-token columns

## v2.0.0 (2026-07-01)

### Highlights
- Complete v2 stack: lexer, parser, type checker, bytecode VM, stdlib
- AI-native design: indentation-based syntax, strong typing, agent protocol
- `Result` + `?` operator for error propagation
- Plan engine with retry, parallel, branch, guard step kinds
- MCP server for LLM integration
- Standard library: json, time, math, str, regex, env, file, http, crypto, jwt, sql

### Features
- **Lexer**: INDENT/DEDENT/NEWLINE, 59 token types, escapes
- **Parser**: Pratt expressions, control flow, function/struct declarations
- **Type System**: 7 type kinds, recursive-descent annotation parser, type checker
- **VM**: typed bytecode, stack-based, frame support, 45 instructions
- **Bytecode Compiler**: 3.5├Ч faster than tree-walking interpreter
- **Stdlib**: json, time, math, str, regex, env, file, http, crypto, jwt, sql
- **Agent Protocol**: meta block, plan block, 6 step kinds, plan engine
- **MCP Server**: 6 tools (ast, format, list_skills, describe_skill, run_skill, lint)

### Limitations (v2.0.x follow-ups)
- 5├Ч interpreter target not yet met (currently 3.5├Ч)
- AI-friendliness benchmark requires community LLM evaluation
- Map literal AST (`{"k": v}`) parser syntax needs explicit braces
- Formatting tool is a no-op (v2.1 will add real formatting)
